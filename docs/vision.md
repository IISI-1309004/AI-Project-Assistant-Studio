### 電梯簡報

> AIPA Studio 是一套可安裝於企業內部的 AI 開發平台。它記住您的架構設計、從每一次 Code Review 中學習，並引導每一個新功能從規格設計走到正式上線——開發人員不需要撰寫任何 Prompt。

---

## 2. AIPA Studio 不是什麼

在定義 AIPA Studio 是什麼之前，必須先明確聲明它不是什麼。以下是不可妥協的非目標。

| 不是這個 | 為什麼這很重要 |
|---|---|
| **不是 ChatBot** | AIPA Studio 不進行對話，而是驅動開發工作流程 |
| **不是 Prompt 工程工具** | 開發人員永遠不需要撰寫 Prompt，系統自動建立所有 AI 上下文 |
| **不是 Copilot Plugin** | AIPA Studio 不是任何單一 AI 工具的延伸，而是協調多個 AI Agent |
| **不是程式碼生成器** | 程式碼生成只是更大、受治理工作流程中的一個小步驟 |
| **不是雲端 SaaS** | AIPA Studio 安裝於企業內部，所有企業知識留在企業內 |
| **不是完全自動化** | 人工審核關卡是強制性的，任何程式碼都不會在未經人工核准下進入正式環境 |

## 3. 問題陳述

### 3.1 根本問題：AI 沒有記憶

現有的 AI 程式碼輔助工具——包括 GitHub Copilot、Cursor 等——都運作在**上下文視窗**之內。每一次 Session 都從零開始。AI 不知道：

- 上個月做了哪些架構決策
- 為什麼選擇了某個設計模式
- 上一個 PR 的 Reviewer 說了什麼
- 團隊的 Coding Style 是什麼
- 哪些業務規則不能被違反
- 什麼東西在正式環境出過問題、原因是什麼

這一類問題，無論多少 Prompt Engineering 都無法從根本上解決。

### 3.2 七大具體痛點

#### 痛點一 — 上下文失憶（Context Amnesia）
AI 只知道當前上下文視窗內的內容。今天修改 A 類別，明天修改 B 類別——AI 不知道 A 與 B 之間存在耦合關係。Regression Bug 被悄悄引入。

**影響**：Regression 比率上升、Code Review 週期浪費、開發人員對 AI 工具產生不信任。

#### 痛點二 — 無架構理解
AI 不知道您的架構。它不知道團隊採用 Hexagonal Architecture、Service 類別不得直接呼叫 Repository、所有資料庫存取必須經過特定的 Transaction Manager。

**影響**：AI 生成的程式碼持續違反架構規則，Reviewer 拒絕 AI 建議，AI 失去公信力。

#### 痛點三 — 無業務規則記憶
AI 不知道業務規則。它不知道付款處理必須同時建立對帳分錄、批次作業不得在月底執行、客戶資料在所有 Log 輸出中必須遮罩。

**影響**：AI 生成的程式碼引入業務邏輯錯誤，這類錯誤代價高昂，在正式環境中更是災難性的。

#### 痛點四 — 無法從歷史中學習
PR Merge 之後，AI 什麼都沒學到。經過 100 次一致性的 Code Review 意見後，AI 仍然犯同樣的錯誤。團隊累積的工程智慧從未以 AI 可使用的形式被保存下來。

**影響**：Code Review 成為永久性的人力消耗，資深工程師離職後，工程知識隨之蒸發。

#### 痛點五 — 缺乏規格紀律
沒有系統強制執行，開發人員往往在需求尚未釐清時就要求 AI 直接 Coding。AI 產出的程式碼滿足了字面上的 Prompt，卻遺漏了實際的業務意圖。

**影響**：程式碼通過單元測試，但在業務驗收時失敗；缺陷在後期才被發現，開發成本增加。

#### 痛點六 — 供應商綁定風險
將開發工作流程建立在單一 AI 供應商（如 GitHub Copilot）上的團隊，會對該供應商的定價、可用性與功能決策產生依賴。

**影響**：失去議價空間、遷移成本高、開發流水線存在單點故障。

#### 痛點七 — 無信心評估機制
AI 從不說「我不知道如何安全地完成這件事」。無論上下文是否充足，它都會生成程式碼。開發人員無法區分高信心的 AI 輸出與純粹猜測的 AI 輸出。

**影響**：風險性變更與安全變更以同等的表面信心被引入；審查負擔均一化，而非依風險比例分配。

## 4. 產品願景

### 願景宣言

> 建立一套具有學習能力的、記憶驅動的企業 AI 開發平台——從每一次專案互動中累積知識，並持續提升每一次程式碼變更的品質與安全性，永無止境。

### 使命宣言

> 透過給予 AI Agent 持久記憶、架構上下文、業務規則知識，以及規格驅動開發的紀律，消除 AI 程式碼能力與企業軟體品質之間的落差。

## 5. 核心哲學：學習規格驅動開發（LSDD）

AIPA Studio 以 **LSDD — Learning Spec Driven Development（學習規格驅動開發）** 作為核心治理哲學。這以結構化的、知識驅動的開發週期取代了臨時性的 Prompt Engineering。

### LSDD 與傳統做法的比較

| 維度 | 傳統 AI Coding | LSDD |
|---|---|---|
| **起點** | 開發人員撰寫 Prompt | 開發人員陳述需求 |
| **上下文** | 塞入上下文視窗的內容 | 所有累積的專案知識與記憶 |
| **規格** | 無，直接 Coding | 正式規格生成並審核後才 Coding |
| **架構** | AI 不知道 | 架構知識內建於系統 |
| **業務規則** | AI 不知道 | 業務規則存於知識引擎 |
| **信心評估** | 永遠 100%（AI 從不拒絕） | 可量測；低於 70% 觸發 NMI |
| **審查** | 可選，純人工 | 強制，自動化 + 人工 |
| **學習** | 無 | 每次 PR Merge 更新知識庫 |
| **AI 供應商** | 綁定單一 AI | Adapter Pattern，支援任意 AI |

### LSDD 十大原則

1. **架構優先（Architecture First）** — 任何功能實作前必須先通過架構審查
2. **規格優先（Specification First）** — 任何程式碼撰寫前必須先有核准的規格文件
3. **知識優先（Knowledge First）** — 所有決策皆以累積的專案知識為依據
4. **記憶優先（Memory First）** — 所有 AI 上下文從持久化的專案記憶中建立，不需人工重新輸入
5. **經驗優先（Experience First）** — 開始新工作前，必定先查閱相似的歷史案例
6. **持續學習（Learning Always）** — 每一個合併的 PR、每一條 Review 意見、每一個決策都被回饋至知識庫
7. **小幅變更（Small Change）** — 所有任務都分解為最小的可獨立測試單元
8. **向下相容（Backward Compatible）** — 所有變更在實作前必須進行影響分析與 Regression 風險評估
9. **測試優先（Test First）** — 測試計劃是規格的一部分，而非事後補充
10. **審查優先（Review First）** — 自動化多維度審查必須在請求人工審查前完成

## 6. 核心能力概覽

### 6.1 知識引擎（Knowledge Engine）
建立並維護專案的結構化知識庫——架構、API、資料庫 Schema、業務規則、工作流程與相依關係。知識隨專案演進持續更新。

### 6.2 記憶引擎（Memory Engine）
跨所有開發 Session 維護持久記憶——Coding Style 決策、架構選擇、Review 模式、團隊慣例與歷史上下文。記憶不隨 Session 結束而消失。

### 6.3 學習引擎（Learning Engine）
自動分析每一個合併的 PR：Git Diff、Commit Message、Reviewer Comment 與業務規則變更。提取模式、決策與知識，回饋至知識庫。每次發布後 AI 都更聰明。

### 6.4 經驗引擎（Experience Engine）
從每一個歷史功能、Bug 修復與重構中建立經驗資料庫。當新需求到來時，系統自動檢索相似歷史案例，用於輔助規格生成與任務規劃。

### 6.5 智慧引擎（Wisdom Engine）
將企業最佳實務與得來不易的教訓編碼為明確規則。例如：「月底不得修改批次作業」、「不得直接修改付款模組」。這些智慧規則在架構審查與規格生成期間被強制執行。

### 6.6 掃描引擎（Scanner Engine）
分析專案程式碼庫，偵測技術棧、架構模式、API、資料庫 Schema、工作流程圖與相依關係樹。支援 Java、Spring Boot、MyBatis、Hibernate、Vue、React、SQL、OpenAPI、Gradle、Maven 等技術。

### 6.7 規格引擎（Specification Engine）
將自然語言需求轉化為正式的結構化規格文件，包含需求細節、知識上下文、記憶引用、影響分析、風險評估、回滾計劃、測試計劃與信心分數。

### 6.8 規劃引擎（Planning Engine）
將規格分解為最小的、可獨立完成的、可追蹤的任務。強制每個任務在下一個開始前獨立完成。

### 6.9 信心引擎（Confidence Engine）
在任何 Coding 開始前評估 AI 的知識涵蓋率。若信心分數低於 70%，Coding 被阻擋，系統發出「需要更多資訊（Need More Information）」請求。防止推測性程式碼生成。

### 6.10 審查引擎（Review Engine）
在每次 AI 生成的變更後執行自動化多維度程式碼審查：架構合規性、Coding 規則合規性、業務規則合規性、安全性、效能、SQL 品質、API 品質與 Regression 風險。

### 6.11 測試引擎（Testing Engine）
為每次生成的程式碼變更自動建立單元測試、整合測試、Regression 測試與 API 測試。測試是規格的一部分，而非事後生成。

### 6.12 AI 介面卡（AI Adapter）
實作 AI Agent 整合的介面卡模式（Adapter Pattern）。支援 GitHub Copilot、Claude Code、Gemini CLI、OpenAI、Ollama 與 MCP。不要求、不優待任何單一 AI 供應商。

## 7. 人工審核關卡模型（Human Checkpoint）

AIPA Studio 並非完全自動化。以下四個關卡強制要求人工核准：

| 關卡 | 時機 | 審核內容 | 核准人員 |
|---|---|---|---|
| **規格核准（Spec Approval）** | 規格引擎生成規格後 | 完整規格：需求、影響、風險、測試計劃 | 開發人員 / 技術負責人 |
| **影響核准（Impact Approval）** | 影響分析顯示高風險變更時 | 影響範圍、受影響元件、回滾計劃 | 技術負責人 / 架構師 |
| **任務核准（Task Approval）** | 規劃引擎完成任務分解後 | 任務清單、相依關係、順序、信心分數 | 開發人員 |
| **PR 核准（PR Approval）** | 審查引擎完成自動審查後 | Code Diff、審查發現、測試結果 | 開發人員 / Reviewer |

審核管道：CLI（主要）、Web UI Dashboard（團隊歷史）、IDE Plugin（通知）。

## 8. 產品架構概要

```
開發人員輸入（一句話需求）
        ↓
  AIPA Runtime Service（Spring Boot，常駐背景服務）
        ↓
  ┌────────────────────────────────────────────┐
  │  知識引擎   │  記憶引擎  │  經驗引擎       │
  └────────────────────────────────────────────┘
        ↓
  規格引擎  →  人工關卡（規格核准）
        ↓
  信心引擎  →  若 < 70%：需要更多資訊
        ↓
  規劃引擎  →  人工關卡（任務核准）
        ↓
  AI 介面卡  →  Copilot / Claude / Gemini / OpenAI / Ollama
        ↓
  程式碼生成  →  測試引擎  →  審查引擎
        ↓
  人工關卡（PR 核准）  →  Git PR  →  Merge
        ↓
  學習引擎  →  知識 / 記憶 / 經驗 更新
        ↓
  （系統對下一個需求更聰明）
```

## 9. 部署模型

AIPA Studio 安裝於**企業內部網路**。所有專案知識、記憶與經驗資料都留在企業邊界內。只有 AI API 呼叫會跨越網路邊界（對 AI 供應商端點的對外 HTTPS 請求）。

支援三種安裝模式：
- **Windows MSI**：單一開發人員工作站，自包含，無外部相依
- **Linux Shell**：團隊伺服器部署，共享知識庫，systemd 管理服務
- **Docker Compose**：容器原生部署，服務編排，企業 DevOps 整合

## 10. 成功標準

### 量化 KPI

| KPI | 現狀（Copilot） | 目標（AIPA Studio） |
|---|---|---|
| AI 生成程式碼的架構違規率 | 約 40% | < 5% |
| AI 生成程式碼的業務規則錯誤率 | 約 30% | < 3% |
| 重複審查意見比率（跨 PR 相同回饋） | 約 60% | < 10% |
| 每次 AI Session 的上下文重新輸入時間 | 15–30 分鐘 | 0 分鐘 |
| 因 AI 程式碼品質導致的 PR 拒絕率 | 約 35% | < 8% |
| 信心 < 70% 阻擋的風險變更率 | 無此機制 | > 95% |
| PR Merge 後知識捕獲率 | 0% | 100% |

### 質化成功標準

1. 開發人員可以輸入一句話，收到一個完整規格化、已審查、已測試的 PR——無需撰寫任何 Prompt
2. 每 10 個合併 PR 後，AI 對專案的知識可量測地提升
3. 新進開發人員僅靠閱讀 AIPA Studio 知識庫即可理解專案架構
4. 資深工程師的智慧被編碼至智慧引擎並自動強制執行
5. 系統可在不變更任何開發工作流程的情況下，從 GitHub Copilot 切換至 Claude Code
